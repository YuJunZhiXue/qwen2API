# 添加工具调用标记清理机制

## 1. 问题背景

### 1.1 核心问题

`##TOOL_CALL##` 和 `[Tool Result]` 标记作为普通文本**永久存储在 Qwen 服务端的对话历史中**。项目使用 chat.qwen.ai 的网页对话接口，关闭了原生函数调用（`function_calling: False`），工具调用以文本标记形式嵌入 prompt。每次复用 `chat_id` 时，这些标记会**越积越多**。

### 1.2 危害

| 影响 | 严重程度 | 说明 |
|------|----------|------|
| 账号封禁 | 🔴 高 | 历史中的标记是 Qwen 官方确定性检测证据（搜索 `##TOOL_CALL##` 即可 100% 识别代理） |
| Token 消耗 | 🟡 中 | 每轮对话额外消耗 14KB+ 的 prompt token |
| 模型混淆 | 🟡 中 | 模型可能尝试模仿历史中的标记格式 |
| 截断风险 | 🟡 中 | 历史过长时早期内容被截断 |

### 1.3 为什么不能简单清洗 prompt 中的标记

整个工具调用链路是**自引用**的——Qwen 通过历史中的 `##TOOL_CALL##` 标记来"学会"输出这种格式。如果清洗掉历史标记：
- 模型失去格式参照，不再输出 `##TOOL_CALL##`
- 少样本注入（`tool_few_shot.py`）失效
- 指令块与历史不一致，模型困惑
- 截断续写（`truncation_recovery.py`）无法检测未闭合标记

**结论**：清洗必须在**会话管理**层面解决，而不是在 prompt 层面。

---

## 2. 解决方案：主动重置 Chat ID

### 2.1 设计思路

不在 prompt 中清洗标记，而是通过**主动创建新 chat_id** 来让 Qwen 服务端从零开始，不积累历史标记。

核心策略：
1. **统计历史标记数**：在 `prompt_builder.py` 中统计历史消息中的 `##TOOL_CALL##` 标记数
2. **截断旧历史**：超过阈值时，只保留 system + 首条 user + 最近 3 轮
3. **强制重置 Chat ID 池**：在 `qwen_executor.py` 中追踪每个 chat_id 的工具调用次数，超过阈值时清空预热池

### 2.2 数据流

```
请求到达
  → prompt_builder.build_prompt_with_tools()
    → 统计历史消息中 ##TOOL_CALL## 标记数
    → 超过阈值？→ 截断旧历史（只保留最近 3 轮）
  → qwen_executor.stream()
    → _check_and_reset_chat_id()
    → 超过阈值？→ chat_id_pool.force_reset() 清空预热池
  → 下次请求创建全新 chat_id → Qwen 服务端无历史标记积累
```

## 3. 实现细节

### 3.1 配置项（`backend/core/config.py`）

新增 3 个环境变量：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `TOOL_CALL_RESET_ENABLED` | `false` | 功能开关 |
| `TOOL_CALL_RESET_THRESHOLD` | `10` | 工具调用次数阈值，超过后强制新建会话 |
| `TOOL_CALL_RESET_CHAT_ID_TTL` | `300` | Chat ID 强制重置后的 TTL（秒） |

### 3.2 历史截断（`backend/services/prompt_builder.py`）

在 `build_prompt_with_tools()` 函数中：

1. 遍历所有历史消息，统计 `##TOOL_CALL##` 标记出现次数
2. 统计范围包括：
   - 字符串内容中的 `##TOOL_CALL##` 文本
   - `tool_use` 类型的消息块
   - `tool_result` 类型的消息块
   - OpenAI 格式的 `tool_calls` 数组
3. 超过阈值时，截断为：system 消息 + 首条 user + 最近 6 条消息（3 轮）
4. 重新构建 `history_parts`，确保预算不超限

### 3.3 Chat ID 追踪与重置（`backend/upstream/qwen_executor.py`）

新增方法和属性：

- `_tool_call_counts: dict[str, int]` — 追踪每个 chat_id 的工具调用次数
- `_count_tool_calls_in_text(text)` — 统计文本中 `##TOOL_CALL##` 标记数
- `_check_and_reset_chat_id(chat_id, prompt_content)` — 检查并触发重置
- `_cleanup_chat_id_tracking(chat_id)` — 清理已结束会话的计数器
- `cleanup_chat_tracking(chat_id)` — 公共接口，请求完成后调用

工作流程：
1. 每次 `stream()` 调用前，统计 prompt 中的标记数
2. 累加到该 chat_id 的计数器
3. 超过阈值时，异步调用 `chat_id_pool.force_reset()`
4. 请求结束后清理计数器

### 3.4 Chat ID 池强制重置（`backend/services/chat_id_pool.py`）

新增 `force_reset(reason)` 方法：
- 清空所有账号的 chat_id 预热队列
- 记录日志（原因、清理数量、账号数）
- 返回清理的总数量

## 5. 使用方式

### 5.1 启用

在 `.env` 中设置：

```env
TOOL_CALL_RESET_ENABLED=true
TOOL_CALL_RESET_THRESHOLD=10
```

### 5.2 调优建议

| 场景 | 建议阈值 | 说明 |
|------|----------|------|
| 高安全要求 | `5` | 更频繁重置，标记积累更少 |
| 平衡模式 | `10` | 默认值，兼顾上下文连续性和安全性 |
| 长任务优先 | `20` | 减少重置频率，保持更多历史上下文 |

### 5.3 日志监控

启用后可通过以下日志关键字监控：

| 关键字 | 含义 |
|--------|------|
| `[Prompt] 工具调用次数=` | 历史截断触发 |
| `[Executor] chat_id=... 工具调用标记数=` | Chat ID 池重置触发 |
| `[ChatIdPool] force_reset` | 预热池清空 |
