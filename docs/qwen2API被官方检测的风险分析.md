# qwen2API 被 Qwen 官方检测的风险分析

## 1. 分析背景

qwen2API 通过在 chat.qwen.ai 网页端和 OpenAI/Claude/Gemini 客户端之间建立协议转换层，让 Claude Code、Cursor 等工具能够调用通义千问模型。本文档分析该架构被 Qwen 官方检测和封禁的风险。

---

## 2. 反检测措施（已有）

| 措施 | 实现文件 | 技术细节 | 有效性 |
|------|----------|----------|--------|
| **Camoufox 反检测浏览器** | `core/browser_engine.py` | 基于 Firefox 内核，自动绕过 bot detection（指纹伪装、Canvas/WebGL 噪声注入、TLS 指纹伪造） | ⭐⭐⭐⭐ 高 |
| **请求抖动** | `core/account_pool/pool_acquire.py` | 随机延迟 120-360ms 模拟真实用户 | ⭐⭐⭐ 中 |
| **会话亲和** | `core/session_affinity.py` | 同一 API Key 复用同一账号，避免频繁切换 | ⭐⭐⭐ 中 |
| **多账号轮询** | `core/account_pool/pool_core.py` | 分散请求到多个账号，降低单账号频率 | ⭐⭐⭐ 中 |
| **HTTP/2 + curl_cffi** | `core/httpx_engine.py` | TLS 指纹伪造，绕过 JA3/JA4 检测 | ⭐⭐⭐⭐ 高 |
| **Chat ID 预热** | `services/chat_id_pool.py` | 复用预建会话，减少 `/api/v2/chats/new` 握手 | ⭐⭐ 低（只减少请求数） |
| **话题隔离** | `services/topic_isolation.py` | 新任务检测，丢弃无关旧历史 | ⭐⭐ 低 |
| **原生函数调用关闭** | `upstream/payload_builder.py` | `function_calling: false`、`enable_tools: false`、`tool_choice: "none"` | ⭐⭐⭐ 中 |

## 3. 可检测的弱点

### 3.1 高风险特征

#### 3.1.1 响应延迟特征异常

| 指标 | 真实用户 | qwen2API 代理 |
|------|----------|---------------|
| 首 token 延迟 | < 1s（本地渲染） | 5~30s（代理转发 + 模型推理 + 解析） |
| 工具调用间隔 | 人类思考时间（秒~分钟） | 几百毫秒（AI 连续调用） |
| 会话时长 | 几分钟~几小时 | 可能持续数小时不间断 |

**检测方式**：如果 Qwen 客户端有埋点统计响应时间分布，代理用户的延迟特征会呈现明显的双峰分布。

#### 3.1.2 请求模式过于规律

真实用户的行为是随机的：浏览、点击、思考、输入、停顿。而 qwen2API 的请求模式是：

```
Read → 0.3s → Write → 0.5s → Bash → 0.2s → Read → 0.4s → Edit → 0.3s → ...
```

一个典型的 Claude Code 任务可能连续调用 20+ 个工具，每个工具请求间隔只有几百毫秒——**这不是人类行为**。

#### 3.1.3 并发特征

- 真实网页端：一个 tab 一个会话
- qwen2API：`MAX_INFLIGHT=2` 意味着同一账号可能同时有 2 个活跃请求
- 多个不同 API Key 的请求映射到同一上游账号 → **同一 IP/设备指纹出现多个独立会话**

### 3.2 中风险特征

#### 3.2.1 工具调用文本标记协议（确定性特征）

**这是最致命的弱点。**

正常 Qwen 网页端对话中，模型输出自然语言。而 qwen2API 的模型被 prompt 引导输出：

```
##TOOL_CALL## {"name":"fs_open_file","input":{"file_path":"/tmp/test.py"}} ##END_CALL##
```

如果 Qwen 服务端做**输出内容审计**，这个标记是**确定性的代理特征**——正常用户永远不会输出这种格式。

**检测难度**：极低。只需在服务端日志中搜索 `##TOOL_CALL##` 字符串即可。

#### 3.2.2 Prompt 注入痕迹

qwen2API 在 prompt 中注入了大量指令：

```
=== ACTION MARKER PROTOCOL (client-parsed text patterns) ===
【重要】用户输入什么语言，就用什么语言回复。
...
WHEN YOU NEED TO TRIGGER AN ACTION — emit this exact text pattern:
##TOOL_CALL##
{"name": "ACTION_NAME", "input": {"param1": "value1"}}
##END_CALL##
...
=== END TOOL INSTRUCTIONS ===
```

如果 Qwen 做 **prompt 审计**（检测 system prompt 注入），这种长指令块很容易被规则匹配或模型分类器识别。

#### 3.2.3 浏览器指纹一致性

- Camoufox 虽然能伪装指纹，但**同一账号长期固定一个指纹**不自然
- 真实用户会更新浏览器、切换设备、清 cookie
- 代理的浏览器指纹可能长期不变或变化模式异常

### 3.3 低风险特征

| 特征 | 风险等级 | 说明 |
|------|----------|------|
| IP 地址 | 低 | 服务器 IP 固定，但很多正常用户也通过 VPN/代理访问 |
| TLS 指纹 | 低 | curl_cffi 的 TLS 指纹伪造已经比较成熟 |
| 请求频率 | 低 | 多账号轮询 + 抖动已有效分散 |
| User-Agent | 低 | Camoufox 自动伪装为真实浏览器 |

## 4. 最可能的检测路径

### 4.1 检测优先级

```
最容易 ←————————————————————————————————————→ 最难

输出内容审计    行为分析    会话分析    Prompt 审计    浏览器指纹
(##TOOL_CALL##)  (高频连续)  (并发会话)  (长指令块)     (固定指纹)
```

### 4.2 检测方式详解

#### 方式 1：输出内容审计（确定性检测）

**原理**：在 Qwen 服务端对模型输出进行文本匹配。

```python
# 伪代码
if "##TOOL_CALL##" in model_output:
    flag_as_proxy(session_id, account_id)
```

**检测率**：接近 100%。每个工具调用都会产生 `##TOOL_CALL##` 标记。

**误报率**：接近 0%。正常用户不会输出这种格式。

#### 方式 2：行为分析（统计检测）

**原理**：统计每个账号的请求频率、间隔分布、会话时长。

```python
# 伪代码
if request_count > 100 and avg_interval < 1.0 and session_duration > 3600:
    flag_as_suspicious(account_id)
```

**检测率**：约 70-80%。需要设置合理的阈值避免误伤。

**误报率**：中等。重度用户可能触发。

#### 方式 3：会话分析（模式检测）

**原理**：检测同一账号的并发会话、IP 切换、设备指纹变化。

```python
# 伪代码
if concurrent_sessions > 1 or unique_ips > 3:
    flag_as_suspicious(account_id)
```

**检测率**：约 50-60%。取决于代理的并发策略。

**误报率**：中等。企业用户可能有多设备登录。

#### 方式 4：Prompt 注入检测

**原理**：检测用户消息中是否包含 prompt 注入特征（长指令块、格式说明等）。

```python
# 伪代码
if "ACTION MARKER PROTOCOL" in user_message or "##TOOL_CALL##" in user_message:
    flag_as_injection(account_id)
```

**检测率**：约 90%。工具结果回传时也会包含 `[Tool Result]` 标记。

**误报率**：低。正常用户不会发送这种内容。

## 5. 封禁风险分级

| 封禁层级 | 手段 | qwen2API 应对难度 | 影响 |
|----------|------|-------------------|------|
| **L1：标记** | 记录包含 `##TOOL_CALL##` 的会话 | 无法避免 | 低（仅标记） |
| **L2：限流** | 对检测到代理特征的账号降速 | 已有冷却机制 | 中（响应变慢） |
| **L3：封账号** | 封禁检测到代理行为的 Qwen 账号 | 需要更换账号 | 高（账号成本） |
| **L4：封 IP** | 封禁服务器 IP 段 | 需要换服务器/代理 | 高（基础设施成本） |
| **L5：改接口** | chat.qwen.ai 网页接口变更 | 需要适配新接口 | 最高（开发成本） |

## 6. 风险评估矩阵

### 6.1 时间维度

| 时间段 | 风险等级 | 原因 |
|--------|----------|------|
| **短期（1-3 个月）** | 🟢 低 | Qwen 官方没有明显动机大规模排查 |
| **中期（3-6 个月）** | 🟡 中 | 如果代理用户增多，官方可能开始审计输出内容 |
| **长期（6 个月+）** | 🔴 高 | `##TOOL_CALL##` 标记是确定性特征，一旦官方想做就能检测 |

### 6.2 最大软肋

**文本标记协议**——模型输出中包含 `##TOOL_CALL##` 是 100% 的代理证据。这是架构层面的缺陷，无法在不改变协议的情况下消除。

### 6.3 最大优势

**Camoufox + curl_cffi 双引擎**——在纯 HTTP 层面的检测（TLS 指纹、浏览器指纹）很难绕过。

---

## 7. 缓解建议

### 7.1 短期缓解

1. **降低请求频率**：增加 `ACCOUNT_MIN_INTERVAL_MS` 到 2000ms+
2. **减少并发**：`MAX_INFLIGHT=1`
3. **定期更换账号**：避免单账号积累过多标记历史

### 7.2 中期缓解

1. **动态文本标记**：不使用固定的 `##TOOL_CALL##`，而是每次随机生成标记
2. **Prompt 混淆**：将长指令块拆分为多个小块，或使用隐写方式嵌入
3. **定期清理历史**：主动创建新 chat_id，丢弃包含标记的旧历史

### 7.3 长期缓解

1. **弃用文本标记协议**：如果 Qwen 官方支持原生 function calling，直接切换
2. **多后端支持**：接入其他模型后端（豆包、智谱等），降低对 Qwen 的依赖
3. **自建模型**：使用开源模型（Qwen 开源版）自建后端，完全避免检测
